# Phase 0 디자인 원칙 구체화 — 토론 결과

## 결론
> Phase 0 Step 0-5를 "아키텍처 원칙"과 "비주얼 디자인 방향" 2파트로 분리. 비주얼 파트는 **디자인 자세, UI 시스템, 앱 쉘+반응형** 3가지만 결정. "취향 설명이 아니라 구현 제약"에 집중. 색상/타이포/아이콘은 design-polish에 위임.

## 합의된 Step 0-5 Part B 구조

### Part B: 비주얼 디자인 방향 (3가지 결정)

Step 0-1의 페르소나 + JTBD + 사용 맥락에서 도출합니다.

overview.md "디자인 원칙" 섹션에 아래 테이블 형식으로 기록:

| 영역 | 결정 | 근거 | 구현 제약 |
|------|------|------|-----------|
| 디자인 자세 | 톤 / 정보 밀도 / 금지 규칙 | 페르소나+JTBD+맥락에서 왜 | Phase 2가 지켜야 할 금지/우선순위 |
| UI 시스템 | 라이브러리 / 커스터마이징 정책 | 일관성, 속도, 접근성 이유 | override 허용 범위 |
| 앱 쉘+반응형 | 내비게이션 / 반응형 / 다크모드 / 전환 규칙 | 사용 환경, 디바이스 비중 | 모바일 전환 규칙 |

#### 1. 디자인 자세 (Design Posture)

페르소나/JTBD/사용 맥락에 맞는 UI 방향을 3요소로 정의:

| 요소 | 설명 | 예시 |
|------|------|------|
| 톤 | 전체적 분위기 | 전문적 / 캐주얼 / 따뜻한 / 냉철한 |
| 정보 밀도 | 화면당 정보량 | 고밀도(대시보드) / 중밀도(SaaS) / 저밀도(랜딩) |
| 금지 규칙 | 이 프로젝트에서 하지 말 것 (2-4개, 관찰 가능한 것만) | "장식용 애니메이션 금지" / "3단 이상 중첩 금지" / "카드 안 카드 금지" |

예: "톤: 전문적, 밀도: 고밀도, 금지: 장식적 애니메이션, 3단 이상 중첩 네비게이션"

금지 규칙 작성 원칙:
- 추상적 금지 불가 ("세련되지 않은 느낌 금지" X)
- 관찰 가능한 것만 ("그라데이션 배경 금지" O)
- 2-4개로 제한

#### 2. UI 시스템

| 결정 | 예시 |
|------|------|
| 컴포넌트 라이브러리 | Tailwind + shadcn/ui, MUI, Ant Design, Chakra UI |
| 커스터마이징 정책 | "라이브러리 기본 primitives 우선, custom variant 최소화" |

#### 3. 앱 쉘 + 반응형

| 결정 | 예시 |
|------|------|
| 내비게이션 구조 | sidebar + content, top-nav + content, bottom-tab(mobile) |
| 반응형 전략 | mobile-first / desktop-first / desktop-only |
| 다크모드 | 지원 / 미지원 / 후속 |
| 핵심 전환 규칙 | "sidebar는 mobile에서 drawer로 축소, 2열은 1열 붕괴, table은 card로 전환" |

### 역할 분리
- **Phase 0**: 구조와 제약 (디자인 자세, UI 시스템, 앱 쉘)
- **design-polish**: 시각 완성도와 세부 미감 (색상 팔레트, 타이포, 아이콘, WCAG)

## 합의된 권장사항

1. Step 0-5를 Part A(아키텍처) + Part B(비주얼) 2파트로 분리
2. Part B는 3가지 결정만 (Phase 0 시간 최소화)
3. overview.md에 4열 테이블(영역/결정/근거/구현 제약)로 기록
4. "안티패턴"을 "금지 규칙"으로 명칭 변경 (더 실행적)
5. 앱 쉘에 "핵심 전환 규칙" 1줄 추가 (반응형 DOM 영향)
6. 색상/타이포/아이콘은 design-polish에 위임

## 토론 과정 요약

### 주요 쟁점
| 쟁점 | codex 입장 | 합의 결과 |
|------|-----------|-----------|
| 디자인 무드 1줄이면 충분? | 너무 약함, 취향이 아닌 구현 제약 필요 | 톤+밀도+금지 규칙 3요소로 구체화 |
