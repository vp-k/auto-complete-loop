# Phase 0 디자인 원칙 구체화 방안

## 현재 상태

pm-planning/SKILL.md Step 0-5 (4줄):
```
프로젝트에 적용할 디자인 원칙/아키텍처 원칙을 정합니다.
기술 스택에 맞는 구체적 원칙 (예: Clean Architecture 레이어, 상태 관리 패턴 등).
```

문제: "디자인 원칙"이라고 하면서 실제로는 아키텍처 원칙만 언급. UI/UX 디자인 방향성이 전혀 없음.

## 목표

Phase 2 구현 시 Claude가 참조할 수 있는 **최소한의 UI/UX 디자인 방향성**을 Phase 0에서 정의.
design-polish/renewal은 구현 후에 적용하는 것으로 확정 - 여기서는 기획 단계 가이드에만 집중.

## PM 관점 질문
1. 디자인 방향이 사용자 페르소나/JTBD와 어떻게 연결되어야 하는가?
2. Phase 0에서 정할 수 있는 디자인 결정의 범위는? (과도하면 Phase 0가 무거워짐)
3. 이 디자인 방향이 Phase 2 구현에 실질적으로 영향을 주려면 어떤 형태여야 하는가?

## 디자이너 관점 질문
1. 프로젝트 유형별 디자인 방향을 어떻게 결정하는가? (SaaS vs 커뮤니티 vs 이커머스 vs 대시보드)
2. Claude가 구현 시 참조할 수 있는 디자인 가이드의 최소 구성 요소는?
3. 색상/타이포/레이아웃/컴포넌트 중 Phase 0에서 결정해야 하는 것과 나중에 polish에서 다듬을 것의 경계는?

## 제약
- output-guard 정책: Phase 0에서 코드/CSS 금지
- Phase 0 전체 시간이 과도하게 늘어나면 안 됨
- design-polish 플러그인과 역할 중복 방지
